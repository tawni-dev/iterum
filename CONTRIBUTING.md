# Contributing to Iterum

Iterum is an AppSec harness for AI-assisted software generation. Changes must preserve its fail-closed authority model and documented security invariants.

## Before contributing

- Open an issue before changing architecture, trust boundaries, pipeline stage ordering, or security invariants.
- Keep `iterum.md`, `pipeline-os.md`, `README.md`, `RESEARCH.md`, and `AUDIT.md` consistent.
- Do not add secrets, credentials, private endpoints, or a local copy of a filled `canon.json`.
- Propose canon value changes in the separate `iterum-pipeline` repository.

## Branches

Use one of these branch formats:

- `feature/<name>`
- `fix/<name>`
- `chore/<name>`
- `security/<name>`

## Pull requests

- Explain what changed and why.
- Identify any security assumptions or trust boundaries affected.
- Update research citations when factual security claims change.
- Record audit findings and resulting remediation where applicable.
- Resolve review comments before merging.
- Do not merge changes generated by an agent without human review.

## Dependencies and licenses

Generated runtime dependencies must be present in the approved pipeline canon and use exact versions without semver ranges. MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause, and ISC dependencies are allowed. GPL or AGPL dependencies require explicit approval.

## Security reports

Do not disclose exploitable vulnerabilities in a public issue. Use GitHub's private vulnerability reporting for the repository.
